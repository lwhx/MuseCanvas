# MuseCanvas MVP 系统设计

## 背景

MuseCanvas 是一个 AI 图像生成平台。当前 MVP 只验证最基础的生图闭环、多个独立用户使用以及平台管理员运维能力，不建设完整创作工作台。

当前仓库尚未包含应用代码或更早序号的功能结果文档。本设计遵循根目录 `AGENTS.md` 规定的技术栈和安全约束。

## 目标

- 管理员可以在开放注册和邀请码注册之间切换，用户全程使用邮箱 OTP 登录，系统不设置密码。
- 用户注册后默认可生图，并且只能访问自己的数据。
- 用户输入提示词、选择管理员启用的模型和基础参数后提交生图任务。
- 生图异步执行，用户可以查看任务状态，并在个人图库中保存和查看每张图片及其提示词。
- 管理员可以管理用户、注册策略、模型启停和全站生成任务，但不能查看用户提示词或图片。
- 密钥只保存在服务端，所有 API 均进行认证、授权和输入校验。
- 服务重启或页面刷新后，已接受的任务仍可恢复和查询。

## 非目标

- MVP 不实现租户、团队、成员邀请或租户角色体系。
- MVP 不实现项目、画布、图层、素材排版或图片编辑器。
- MVP 不实现 LLM 提示词优化、提示词模板或工作流编排。
- MVP 不实现额度、套餐、支付、账单或商业化计费。
- MVP 不实现实时协作、公开作品社区、评论或推荐。
- MVP 不实现图生图、局部重绘、模型训练或 LoRA 管理。
- MVP 不实现密码登录、密码找回或第三方社交登录。
- MVP 不实现图像收藏、标签、文件夹、搜索或图库分享；图库现阶段只负责存储和展示图片与提示词。

## 术语

- **用户（User）**：使用平台生成和管理自己图片的普通账户。
- **管理员（Admin）**：拥有平台级管理权限的账户，不属于单独租户。
- **生成任务（Generation Job）**：一次异步执行的文生图请求。
- **资产（Asset）**：生图成功后保存的图片及其元数据。
- **模型配置（Model Configuration）**：管理员维护的可用模型、参数范围和启用状态。
- **OTP**：发送至用户邮箱的一次性验证码，用于注册和登录。
- **邀请码（Invitation）**：邀请注册模式下，由管理员签发并绑定目标邮箱的单次注册凭证。
- **图库（Library）**：当前用户私有的生成图片集合，每张图片保存对应提示词。

## 产品范围

### 用户端

1. 用户先提交邮箱。已注册邮箱直接发送 OTP；开放注册模式下的未注册邮箱也直接发送 OTP；邀请码模式下的未注册邮箱先返回 `registration_required`，不发送 OTP。
2. 收到 `registration_required` 后，前端进入邀请码步骤；用户提交与邮箱绑定的有效邀请码后，服务端才发送 OTP 并进入验证码步骤。
3. 用户输入 OTP 完成注册或登录；新注册用户状态默认为 `active` 并可立即生图。
4. 用户进入生图页，输入提示词。
5. 用户选择管理员已启用的模型、尺寸、质量和生成张数等该模型支持的基础参数。
6. 服务端验证身份、输入和模型配置，创建生成任务。
7. Worker 异步调用真实生图供应商并保存结果。
8. 前端轮询任务状态，展示排队、生成中、成功或失败。
9. 每张成功图片和对应提示词自动进入用户私有图库，用户可查看、下载或软删除。

### 管理员后台

管理员后台与普通用户页面共用前端工程，但使用独立路由和服务端权限检查。MVP 包含：

- 仪表盘：用户总数、任务总数、近期成功率和失败数。
- 注册管理：切换 `open` 或 `invite_only` 注册模式，创建、查看和撤销邀请码。
- SMTP 管理：配置服务器、端口、发件人、用户名以及 SSL/TLS 模式，写入或轮换 SMTP 凭据并发送测试邮件。
- 用户管理：分页查询用户、查看状态、停用、恢复或软删除用户。
- 模型管理：按 Image-API 文档手动添加获批模型，配置模型参数和并发上限，并逐个启用或关闭。
- 任务管理：按用户、状态、模型和时间查询任务，只查看任务 ID、状态、模型、时间和过滤后的错误码。

管理员不能查看用户 OTP、会话 Token、供应商密钥、提示词、图片、对象存储地址或未经过滤的供应商响应。管理 API 的任务 DTO 和数据查询不得选择提示词及资产字段；排障只使用任务 ID、错误码、模型、耗时和供应商引用 ID。

## 权限模型

MVP 只有两个平台角色：

| 角色 | 权限 |
|---|---|
| `user` | 创建任务，查询和管理自己的任务及资产 |
| `admin` | 使用管理员后台管理用户、模型和全站任务 |

- 角色来自服务端会话，不接受前端请求体传入的角色或用户 ID。
- 普通用户资源查询必须包含 `created_by = current_user_id`。
- 管理 API 必须单独执行 `admin` 角色校验，不能只依赖前端隐藏菜单。
- 禁用用户后，其现有会话应立即或在很短时间内失效，且不能继续提交任务。
- 软删除用户时立即撤销会话、取消尚未执行的任务，并将用户及其图库标记删除；对象文件由定时任务统一异步清理。
- 首个管理员通过受控的部署初始化流程创建，禁止开放注册参数将自己提升为管理员。

本 MVP 没有租户概念，因此不使用 `tenant_id`。若后续引入团队空间，应在独立功能设计和 migration 中增加租户模型，不能把用户 ID 临时当成租户 ID。

## 总体架构

采用 pnpm workspace 管理的模块化单体和独立 Worker：

```text
Browser
  |
  v
Nginx
  |-- /        -> Vue 3 + Vite 静态资源
  `-- /api/*   -> Next.js API Routes
                       |-- PostgreSQL（用户、任务、资产元数据）
                       |-- Redis（任务队列、限流、短期状态）
                       |-- S3 兼容对象存储（图片二进制）
                       |-- SMTP Server（发送 OTP）
                       `-- Image Provider API

Worker
  |-- 消费 Redis Stream
  |-- 调用 Image Provider API
  |-- 写入对象存储
  `-- 更新 PostgreSQL 任务和资产
```

建议目录：

```text
apps/
  web/                 # Vue 用户端和管理员后台
  api/                 # Next.js API Routes
  worker/              # TypeScript 生图 Worker
packages/
  contracts/           # API DTO、共享类型和错误码
  domain/              # 权限、任务状态机等业务规则
  database/            # migration、事务和数据访问
  providers/           # 生图和对象存储适配器
  config/              # 服务端环境变量校验
infra/
  nginx/
  docker/
```

`contracts` 只共享允许返回浏览器的字段，不共享数据库实体或供应商原始响应。

## 模块设计

### 认证与用户

- 系统没有密码字段，也不提供密码登录或找回入口；邮箱 OTP 是唯一用户认证方式。
- `POST /api/auth/otp/request` 接收邮箱和可选邀请码，并返回明确的下一步：`otp` 或 `invitation`。
- 已注册邮箱返回 `otp` 并发送验证码；开放注册模式下的未注册邮箱同样返回 `otp`；邀请码模式下的未注册邮箱在未提交邀请码时返回 `invitation`，且不创建 OTP challenge、不发送邮件。
- 邀请码步骤提交无效、过期、已使用、已撤销或邮箱不匹配的邀请码时返回稳定错误码 `INVALID_INVITATION`；验证有效后才创建 OTP challenge 并返回 `otp`。
- 该交互会在邀请码模式下暴露邮箱是否已注册，这是已确认的产品取舍；OTP 请求仍必须同时按邮箱和 IP 限流，响应不得暴露用户角色、状态、用户 ID 或其他账户信息。
- OTP 使用密码学安全随机数生成，只保存验证码哈希、用途、过期时间、失败次数和消费时间；验证码短时有效且只能成功使用一次。
- OTP 的发送频率按邮箱和 IP 限流，验证尝试次数有上限；重新发送后旧验证码失效。
- `open` 模式下，验证未注册邮箱的 OTP 后自动创建 `active` 用户；`invite_only` 模式下还必须原子校验并消费与邮箱匹配的有效邀请码。
- 邀请码只向用户展示一次，数据库保存哈希；邀请码绑定邮箱、有效期和单次使用状态，管理员可以在使用前撤销。
- 浏览器使用安全会话 Cookie；Cookie 设置 `HttpOnly`、`Secure` 和合适的 `SameSite`。
- 用户状态包含 `active`、`disabled`，软删除由 `deleted_at` 表示。
- 停用、软删除账户或管理员重置安全状态时，撤销全部既有会话。
- 个人资料 MVP 只包含必要字段，避免收集与生图无关的信息。

OTP 邮件通过管理员配置的 SMTP 服务发送，支持隐式 TLS/SSL 和 STARTTLS。配置包含主机、端口、发件地址、发件名称、用户名、TLS 模式和凭据：

- SMTP 凭据只允许写入或轮换，任何读取 API 均返回“已配置”状态而不返回原文。
- SMTP 凭据使用仅存在服务端环境变量中的主密钥加密后入库；主密钥不得存入数据库、前端或日志。
- 连接测试由后端执行，只返回成功或经过过滤的错误码，不回传底层 SMTP 错误和凭据。
- 生产环境禁止关闭传输加密；证书校验默认开启，不能通过后台绕过。
- SMTP 配置变更和测试发送均写入审计日志，但不记录凭据或完整 OTP 邮件内容。

### 基础生图

MVP 根据 `Image-API/GPT-IMAGE-2_API.md` 和 `Image-API/SEEDREAM_API.md` 接入 GPT Image 2 与 Seedream，但只提供文生图。统一输入包含：

- `prompt`：必填，限制长度并过滤不可接受内容。
- `model_id`：必须引用管理员已启用的模型。
- `size`：必须属于所选模型允许值；GPT Image 2 使用 `1024x1024`、`1536x1024`、`1024x1536`，Seedream 使用管理员为具体模型配置的 `2K`、`4K` 或合法固定尺寸。
- `count`：限制在管理员配置和系统上限内。
- `quality`：仅 GPT Image 2 使用，值限制为 `low`、`medium`、`high` 或 `auto`。

平台固定使用非流式调用。GPT Image 2 固定输出 PNG 并接收 base64；Seedream 固定 `response_format: "url"`、`stream: false` 和管理员配置的水印策略。API 不接受供应商 URL、密钥、任意模型名称或未经声明的参数。参考图、图像编辑、组图、流式输出及任意高级参数均不进入 MVP。

### 模型配置

- 管理员根据供应商实际授权手动添加模型。系统不自动探测或假定账户已获批任何模型。
- 数据库保存模型展示名称、`openai` 或 `seedream` 适配器类型、供应商模型标识、支持尺寸、质量选项、最大生成张数、水印策略、并发上限、启用状态和排序。
- 供应商密钥只来自环境变量，不写入模型配置表、不通过管理 API 返回。
- 禁用模型后不能创建新任务，但历史任务仍能正确显示原模型快照。
- 任务创建时保存模型名称和参数快照，避免管理员修改配置导致历史记录含义变化。
- MVP 实现 GPT Image 2 和 Seedream 两个 `ImageProvider` 适配器，不建设动态插件系统。
- OpenAI 适配器调用 `POST /v1/images/generations`，从 `b64_json` 解码图片后转存。
- Seedream 适配器调用 `${ARK_BASE_URL}/api/v3/images/generations`，收到短期 URL 后立即由 Worker 下载并转存。
- 供应商模型标识来自管理员配置；业务代码不得硬编码 Seedream 的版本号。
- Worker 按模型配置的并发上限在 Redis 中获取带租约的分布式许可；没有许可的任务保持排队，不超额调用供应商。
- Worker 异常退出后许可随租约过期自动释放；任务完成、失败或取消时主动释放。
- 并发配置变更只影响后续许可获取，不强行中断正在执行的供应商请求。

### 异步任务

任务状态机：

```text
queued -> running -> succeeded
                  `-> failed
       `-> canceled
running -> retry_wait -> queued
```

- API 在一个 PostgreSQL 事务内创建任务和 outbox 事件。
- Dispatcher 将 outbox 事件投递到 Redis Stream，成功后记录投递状态。
- Worker 使用消费者组、任务租约和幂等键，防止重复消费产生重复资产。
- PostgreSQL 任务记录是权威状态；Redis 丢失后可通过未完成任务和 outbox 恢复。
- 临时网络错误有限重试，参数或内容审核等永久错误不重试。
- 前端轮询 `GET /api/jobs/:id`；页面刷新后从数据库恢复，不依赖长连接。
- 供应商错误转换为平台错误码，不能原样返回内部请求、响应或密钥。

部分供应商不保证幂等提交。Worker 应保存供应商引用 ID，超时后优先查询已有任务；无法判断是否已提交时不得直接重复调用。

### 资产管理

MVP 将资产管理对用户呈现为“图库”。现阶段不做组织和编辑能力，只保证每张成功图片与生成它的用户提示词一起持久化。

- PostgreSQL 为每张输出图片保存所有者、对应提示词、对象键、MIME 类型、尺寸、大小、校验值、来源任务和生成时间。
- 即使一次任务返回多张图片，每张图片也分别建立资产记录并保存同一提示词快照。
- 图片二进制放入对象存储，不能放入 PostgreSQL 或 Redis。
- 对象键使用不可猜测 UUID，不包含邮箱、用户名或提示词。
- 图库列表和下载均校验当前用户是资产所有者；管理员角色不获得跨用户图库读取或下载权限。
- 删除先写入 `deleted_at` 并创建持久化清理任务，后台定时任务统一删除 S3 对象。
- GPT Image 2 的 base64 在 Worker 内解码；Seedream 的临时 URL 由 Worker 立即下载。两者均在校验文件类型、大小和图片尺寸后上传对象存储。

对象存储固定支持 S3 兼容协议。Endpoint、Region、Bucket、Access Key、Secret Key 和签名地址有效期通过服务端环境变量配置；凭据不进入管理员后台。数据库只保存对象键，不保存永久公开 URL。下载必须在鉴权后生成短期签名 URL。

### 用户删除清理

- 管理员删除用户时只做软删除：设置用户 `deleted_at` 和 `deletion_requested_at`，立即撤销全部会话并阻止 OTP 登录。
- 同一事务创建持久化 `deletion_jobs` 记录，避免服务重启后遗漏清理。
- 定时任务批量处理待清理用户，将其任务和资产标记软删除，并删除对应 S3 对象。
- 清理成功后记录 `completed_at`；部分失败时保留失败对象键和重试次数，后续继续重试，不能把失败任务伪装成完成。
- 数据库保留用户、任务和资产的软删除墓碑及必要审计关联，不物理删除核心记录；提示词和邮箱等用户内容在清理完成后清空或不可逆匿名化。
- 清理流程不向管理员展示图片、提示词或对象内容，只处理内部 ID 和对象键。

### 管理员审计

- 注册模式变更、邀请码创建/撤销、用户停用/恢复/软删除、模型创建/修改/启停均写入审计日志。
- 审计记录包含管理员 ID、操作类型、目标类型、目标 ID、时间和必要的变更摘要。
- 审计日志不记录 OTP、Token、邀请码原文、提示词、图片地址、供应商密钥或完整供应商响应。
- MVP 后台可以不提供审计日志页面，但数据必须落库，为后续追责保留依据。

## 数据模型

所有主键使用 UUID，时间字段使用带时区时间戳。用户归属数据包含 `created_by`、`created_at`、`updated_at` 和适用时的 `deleted_at`。

| 表 | 用途 | 关键约束 |
|---|---|---|
| `users` | 无密码用户账户和平台角色 | 邮箱唯一，角色只能由管理员流程修改，新用户默认 `active` |
| `sessions` | 可撤销会话 | 保存哈希后的会话标识和过期时间 |
| `otp_challenges` | OTP 请求和验证状态 | 只保存 OTP 哈希，短时有效且单次消费 |
| `registration_settings` | 全局注册策略 | 单例记录，模式为 `open` 或 `invite_only` |
| `invitations` | 邀请注册凭证 | 邀请码哈希唯一，绑定邮箱、有效期和消费状态 |
| `smtp_settings` | 管理员维护的 OTP 邮件配置 | SMTP 凭据加密保存且 API 永不回显 |
| `model_configs` | 管理员手动维护的模型能力 | 供应商密钥不入表，并发上限为正整数 |
| `generation_jobs` | 生成任务权威状态 | `(created_by, idempotency_key)` 唯一 |
| `generation_outputs` | 任务与资产关联 | `(job_id, asset_id)` 唯一 |
| `assets` | 私有图库中的图片和提示词 | 对象键唯一，所有者及提示词快照必填 |
| `outbox_events` | 可靠队列投递 | 事件 ID 唯一，可重试 |
| `deletion_jobs` | 用户和资产的定时清理状态 | 每个删除目标仅有一个活动清理任务 |
| `audit_logs` | 管理操作审计 | 只追加，不含敏感值 |

任务和资产查询不得提供省略 `created_by` 的普通用户数据访问方法。管理员任务查询使用专用的诊断投影，SQL 和 DTO 均不选择 `prompt`、对象键、图片地址或输出资产关联。

## API 设计

统一响应格式：

```json
{ "success": true, "data": {} }
```

```json
{
  "success": false,
  "error": { "code": "FORBIDDEN", "message": "无权执行该操作" }
}
```

### 公开与用户 API

| 方法与路径 | 用途 |
|---|---|
| `POST /api/auth/otp/request` | 提交邮箱并获取下一步；返回 `otp` 或 `invitation`，邀请码验证通过后才发送注册 OTP |
| `POST /api/auth/otp/verify` | 验证 OTP，按当前注册策略注册或登录 |
| `POST /api/auth/logout` | 退出并撤销当前会话 |
| `GET /api/session` | 获取当前用户和角色摘要 |
| `GET /api/models` | 查询已启用模型及允许参数 |
| `POST /api/generations` | 创建当前用户的生成任务 |
| `GET /api/jobs` | 分页查询当前用户任务 |
| `GET /api/jobs/:id` | 查询当前用户单个任务及输出 |
| `POST /api/jobs/:id/cancel` | 取消尚可取消的当前用户任务 |
| `GET /api/library` | 查询当前用户的图片与提示词 |
| `GET /api/library/:id/download` | 获取当前用户图片的短期授权下载地址 |
| `DELETE /api/library/:id` | 软删除当前用户图库图片 |

### 管理 API

| 方法与路径 | 用途 |
|---|---|
| `GET /api/admin/dashboard` | 查询管理指标摘要 |
| `GET /api/admin/users` | 分页查询用户 |
| `PATCH /api/admin/users/:id/status` | 禁用或恢复用户 |
| `DELETE /api/admin/users/:id` | 软删除用户并撤销其会话 |
| `GET/PATCH /api/admin/registration` | 查询或切换开放/邀请码注册模式 |
| `GET/POST /api/admin/invitations` | 查询或创建邀请码；列表不返回邀请码原文 |
| `DELETE /api/admin/invitations/:id` | 撤销未使用邀请码 |
| `GET/PUT /api/admin/smtp` | 查询脱敏配置或整体更新 SMTP 配置 |
| `POST /api/admin/smtp/test` | 使用已保存配置发送测试邮件 |
| `GET/POST /api/admin/models` | 查询或创建模型配置 |
| `PATCH /api/admin/models/:id` | 修改、启停模型配置 |
| `GET /api/admin/jobs` | 查询不含提示词和图片字段的任务诊断信息 |

所有输入执行运行时校验，分页采用有上限的游标。写接口接收客户端幂等键，重复请求返回原任务。错误响应只返回稳定错误码和安全消息，内部日志通过 `request_id` 关联。

## 前端设计

### 用户端路由

- `/login`：统一 OTP 注册和登录页面，使用 `email -> invitation（仅需要时）-> otp` 三步状态机；已注册用户不显示邀请码步骤。
- `/generate`：提示词、模型和参数表单，显示当前任务状态。
- `/library`：当前用户的图片与提示词列表。
- `/account`：最小账户信息和退出入口。

### 管理后台路由

- `/admin`：指标摘要。
- `/admin/registration`：注册模式和邀请码管理。
- `/admin/smtp`：SMTP SSL/TLS 配置和测试邮件。
- `/admin/users`：用户管理。
- `/admin/models`：模型管理。
- `/admin/jobs`：任务查询。

Vue Router 导航守卫根据会话改善体验，但服务端始终重新鉴权。Pinia 只保存会话摘要、表单临时状态和任务轮询状态。组件优先使用 shadcn-vue / Reka UI，样式只使用 Tailwind CSS。

轮询使用退避和页面可见性控制；任务进入终态后停止，网络恢复或页面刷新时重新读取服务端状态。

## 安全与隐私

- 所有生成、资产和管理接口必须认证；管理接口额外校验 `admin` 角色。
- OTP 请求与验证、生图提交、任务查询和下载签名均按用户、邮箱或 IP 限流。
- 状态变更接口防护 CSRF。
- 提示词需进行输入安全检查，供应商返回图片需执行输出安全策略。
- 日志过滤 Cookie、Token、OTP、邀请码、供应商密钥、提示词和图片地址。
- `.env.example` 只包含变量名和非敏感说明，不提供真实凭据。
- SMTP 加密主密钥、S3 凭据和供应商 API Key 只存在服务端环境变量；SMTP 凭据在数据库中使用主密钥加密。
- API 输出使用明确 DTO，禁止直接序列化数据库用户实体。
- 管理 API 不复用用户图库 DTO；管理员查询链路从数据库投影开始就排除提示词和图片字段。

## 可观测性与恢复

- 每个 API 请求生成 `request_id`，生成任务使用 `job_id` 贯穿日志、队列和供应商调用。
- 关键指标包括注册/登录失败率、API 错误率、队列积压、任务耗时、成功率、供应商错误率和重试次数。
- 定时任务负责重投 outbox、恢复失去租约的任务、处理用户删除任务和统一清理已软删除的 S3 对象。
- PostgreSQL 和对象存储是备份核心；Redis 丢失不应造成已接受任务永久丢失。
- 健康检查区分进程存活和依赖就绪。

## 部署设计

Docker Compose 至少包含：

- `web`：Vue 静态资源构建。
- `api`：Next.js API Routes。
- `worker`：生图任务、outbox 投递和定时删除清理进程。
- `postgres`：权威数据库。
- `redis`：Stream、限流和短期缓存。
- `nginx`：反向代理和静态资源服务。

对象存储统一使用 S3 兼容协议，本地可运行 S3 兼容服务，生产环境可使用托管 S3 服务。不同进程使用最小权限环境变量，Web 构建不得获得数据库、SMTP 加密主密钥、S3 或供应商凭据。

## 测试策略

- 单元测试覆盖 OTP 有效期/次数、注册模式、邀请码消费、SMTP 配置脱敏、角色授权、用户状态、任务状态机、模型参数校验、并发许可和错误映射。
- 数据库集成测试覆盖邀请码并发消费、用户数据隔离、任务幂等、软删除和 outbox 一致性。
- API 测试覆盖邮箱下一步分支、邀请码未通过时不发送 OTP、OTP 限流、未登录访问、普通用户越权、管理员访问和停用/删除账户会话失效。
- Worker 分别覆盖 GPT Image 2 base64 转存、Seedream 临时 URL 转存、模型并发租约、定时删除及失败重试，以及超时、重复投递和供应商错误；测试替身只用于测试。
- 端到端测试覆盖开放注册、邀请码注册、OTP 登录、提交真实配置任务、轮询结果、图库查看和管理员停用用户。
- 每个用户资源接口至少验证：所有者可访问、其他用户不可访问、软删除后不可访问。
- 管理 API 契约测试必须断言响应中不存在提示词、对象键、图片 URL 或图片内容。

## 分阶段交付

### 阶段 1：账户和管理骨架

- workspace、环境配置、Docker Compose、Nginx 和 migration。
- OTP 注册/登录、会话、`user/admin` 授权和首个管理员初始化。
- 开放/邀请码注册切换、邀请码管理、SMTP SSL/TLS 配置及审计记录。
- 用户停用/软删除、持久化删除任务和定时清理。
- GPT Image 2 与 Seedream 模型手动添加、启停和并发配置。

### 阶段 2：真实生图闭环

- 按 `Image-API` 文档实现 GPT Image 2 和 Seedream 文生图适配器。
- 任务、outbox、Redis Stream 和 Worker。
- S3 兼容对象存储、逐图保存提示词、用户生图页和状态轮询。

### 阶段 3：图库与运维

- 用户私有图库、下载和软删除。
- 管理任务查询、仪表盘指标、限流和修复任务。
- 完整权限测试、故障恢复测试和部署检查。

只有真实供应商调用、任务恢复、图片持久化和用户数据隔离全部通过，才算完成 MVP；不能用 mock 数据替代生产闭环。

## 已确认决策

1. 用户认证只使用邮箱 OTP；管理员通过后台配置支持 SSL/TLS 的 SMTP 服务。
2. 管理员根据实际授权手动添加 GPT Image 2 或 Seedream 模型，并配置启用状态、参数和并发上限。
3. 图片使用 S3 兼容对象存储，数据库仅保存对象键和元数据。
4. 用户删除采用软删除和持久化清理任务，由定时任务统一清理 S3 对象并匿名化用户内容。

## 验收标准

- 未登录用户不能访问生图、图库或管理接口。
- 管理员可在开放注册和邀请码注册间切换；邀请码模式下未注册邮箱必须先通过邀请码步骤才能接收 OTP，邀请码只能被绑定邮箱使用一次；已有用户直接接收 OTP。
- 系统不存在密码注册、密码登录和密码找回路径，OTP 不以明文持久化且不可重复使用。
- 管理员可配置 SMTP SSL/TLS 并发送测试邮件，管理 API 永不返回 SMTP 凭据原文。
- 普通用户无法通过猜测 ID、列表参数或下载地址访问其他用户的任务和图片。
- 普通用户无法调用任何管理 API；管理员权限由服务端会话确定。
- 新注册用户默认可生图；被停用或软删除的用户不能继续使用旧会话提交或读取任务。
- 同一用户以同一幂等键重复提交时，只创建一个任务。
- API、Worker 或 Redis 重启后，已接受任务能恢复并进入明确终态。
- 生图成功后图片持久化并可授权下载；失败时返回安全、可理解的错误码。
- GPT Image 2 的 base64 和 Seedream 的临时 URL 均被转存为平台资产，不向前端持久暴露供应商原始结果。
- 管理员可手动添加获批模型并设置并发上限；多 Worker 同时运行时实际调用数仍不超过该上限。
- 每张成功图片及其提示词都保存在所有者私有图库中。
- 用户删除后立即失去访问权限，定时任务最终删除关联 S3 对象；清理失败可重试且状态可审计。
- 管理员能切换注册模式、管理邀请码、停用/恢复/软删除用户、启停模型，并查询全站任务的非敏感诊断信息。
- 管理员页面、管理 API 和管理员专用数据查询均不能返回用户提示词或图片。
- 数据库异常、供应商原始错误和所有密钥均不会返回前端或进入公开日志。
