# 文档工作流规范

> 仅适用于 `/docs` 目录。所有功能文档按编号文件夹组织，遵循本文档约定。

## 文件夹命名

每个功能一个文件夹，格式：`{序号}-{功能名}`

```
docs/
├── 001-mvp/
├── 002-额度系统/
├── 003-多租户/
└── ...
```

- 序号：三位数字，递增
- 功能名：简短的中英文均可，用短横线分隔
- 文件夹内文件全部小写，用短横线分隔

## 文档类型与生命周期

一个功能的完整文档链路：

```
design.md → impl.md → result.md
 (设计)      (实现)     (结果)
```

| 文件 | 何时生成 | 说明 |
|---|---|---|
| `design.md` | 需求讨论阶段 | 功能设计方案，明确目标和边界 |
| `impl.md` | 编码开始前 | 实现计划，拆分任务和依赖 |
| `result.md` | 功能完成后、最终回复前 | 实际实现的总结和验收依据；缺失时不得宣告功能完成 |

### design.md（设计文档）

每次和用户商议新功能时生成。

```markdown
# {功能名}设计

## 背景

基于前置文档（仅参考序号 < 本文件夹序号 的文档），描述当前项目状态。
重点关注对应目录下的 result.md，了解已有实现。

## 目标

- 本次功能要实现的内容
- 逐条列出

## 非目标

- 本次不实现的内容
- 避免范围蔓延

## 术语

- 专业术语及解释
- 便于后续文档保持一致
```

### impl.md（实现计划）

编码前将设计拆解为可执行的任务。

```markdown
# {功能名}实现计划

## 设计文档

引用：`../{序号}-{功能名}/design.md`

## 任务拆解

- [ ] 任务 1：描述
- [ ] 任务 2：描述（依赖任务 1）
- [ ] ...

## 涉及文件

列出需要新增或修改的文件路径和原因。

## 注意事项

- 已知风险或需要确认的决策点
```

### result.md（结果记录）

功能达到终止条件后立即创建或更新，记录实际做了什么。`result.md` 是功能完成的强制交付物，不是可选总结；Agent 必须在向用户发送完成答复前确认它存在并与当前实现一致。

```markdown
# {功能名}实现结果

## 设计文档

引用：`design.md`

## 结果状态

- 状态：已完成 / 部分完成 / 未完成
- 完成日期：YYYY-MM-DD

## 完成情况

- [x] 任务 1：已完成
- [x] 任务 2：已完成
- [ ] 任务 3：未完成，原因

## 变更摘要

简述实际改动了哪些文件、新增了哪些内容。

## 验证记录

- `pnpm typecheck`：通过
- `pnpm test`：通过
- 未执行项：说明原因和影响

## 设计偏差

- 记录实现与 `design.md` 不一致的部分及原因；没有则写“无”

## 遗留问题

- 未完成的部分
- 后续需要优化的点
- 发现的 bug 或技术债
```

## 引用规则

- 设计文档只能引用**序号更小**的文件夹内容（避免循环依赖）
- 引用格式：`../{序号}-{功能名}/result.md`
- 新功能的设计必须先阅读已有 result.md，了解当前项目状态

## Agent 操作规则

1. **讨论阶段** → 仅生成 `design.md`
2. **编码前** → 补充 `impl.md`
3. **完成后、最终回复前** → 创建或更新 `result.md`，并检查内容与实际代码、测试和部署状态一致
4. 修改文档时，保留原有内容，在此基础上追加或修订
5. 不得删除或覆盖其他功能的文档
6. 不得在缺少最新 `result.md` 时向用户宣告功能完成；若验证未通过或外部依赖未接通，必须将状态写为“部分完成”并如实记录
7. 完成后的任何代码修复、验收补充或设计变更，都必须同步更新该功能的 `result.md`
8. 若历史功能缺少 `impl.md`，不得事后伪造编码前计划；在 `result.md` 中说明历史缺失，并保证后续功能严格执行完整链路

## 完成前强制检查

Agent 在结束一个编号功能前必须逐项确认：

- [ ] 同目录存在 `design.md`
- [ ] 同目录存在 `impl.md`；历史缺失已在 `result.md` 如实说明
- [ ] 同目录存在 `result.md`
- [ ] `result.md` 覆盖实际完成项、验证记录、设计偏差和遗留问题
- [ ] 最终回复中的完成状态与 `result.md` 一致

## 示例工作流

```
用户：我们来讨论额度系统的设计
Agent：已创建 docs/002-额度系统/design.md，请查看

用户：设计确认，开始实现
Agent：已创建 docs/002-额度系统/impl.md，拆分为 5 个任务

用户：额度系统完成了
Agent：已更新 docs/002-额度系统/result.md，记录完成情况
```
